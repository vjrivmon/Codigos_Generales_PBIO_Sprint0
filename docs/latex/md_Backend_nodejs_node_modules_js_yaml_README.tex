\hypertarget{md_Backend_nodejs_node_modules_js_yaml_README_autotoc_md1647}{}\doxysubsection{load (string \mbox{[} , options \mbox{]})}\label{md_Backend_nodejs_node_modules_js_yaml_README_autotoc_md1647}
Parses {\ttfamily string} as single YAML document. Returns either a plain object, a string, a number, {\ttfamily null} or {\ttfamily undefined}, or throws {\ttfamily YAMLException} on error. By default, does not support regexps, functions and undefined.

options\+:


\begin{DoxyItemize}
\item {\ttfamily filename} \+\_\+(default\+: null)\+\_\+ -\/ string to be used as a file path in error/warning messages.
\item {\ttfamily on\+Warning} \+\_\+(default\+: null)\+\_\+ -\/ function to call on warning messages. Loader will call this function with an instance of {\ttfamily YAMLException} for each warning.
\item {\ttfamily schema} \+\_\+(default\+: {\ttfamily DEFAULT\+\_\+\+SCHEMA})\+\_\+ -\/ specifies a schema to use.
\begin{DoxyItemize}
\item {\ttfamily FAILSAFE\+\_\+\+SCHEMA} -\/ only strings, arrays and plain objects\+: \href{http://www.yaml.org/spec/1.2/spec.html\#id2802346}{\texttt{ http\+://www.\+yaml.\+org/spec/1.\+2/spec.\+html\#id2802346}}
\item {\ttfamily JSON\+\_\+\+SCHEMA} -\/ all JSON-\/supported types\+: \href{http://www.yaml.org/spec/1.2/spec.html\#id2803231}{\texttt{ http\+://www.\+yaml.\+org/spec/1.\+2/spec.\+html\#id2803231}}
\item {\ttfamily CORE\+\_\+\+SCHEMA} -\/ same as {\ttfamily JSON\+\_\+\+SCHEMA}\+: \href{http://www.yaml.org/spec/1.2/spec.html\#id2804923}{\texttt{ http\+://www.\+yaml.\+org/spec/1.\+2/spec.\+html\#id2804923}}
\item {\ttfamily DEFAULT\+\_\+\+SCHEMA} -\/ all supported YAML types.
\end{DoxyItemize}
\item {\ttfamily json} \+\_\+(default\+: false)\+\_\+ -\/ compatibility with JSON.\+parse behaviour. If true, then duplicate keys in a mapping will override values rather than throwing an error.
\end{DoxyItemize}

NOTE\+: This function {\bfseries{does not}} understand multi-\/document sources, it throws exception on those.

NOTE\+: JS-\/\+YAML {\bfseries{does not}} support schema-\/specific tag resolution restrictions. So, the JSON schema is not as strictly defined in the YAML specification. It allows numbers in any notation, use {\ttfamily Null} and {\ttfamily NULL} as {\ttfamily null}, etc. The core schema also has no such restrictions. It allows binary notation for integers.\hypertarget{md_Backend_nodejs_node_modules_js_yaml_README_autotoc_md1648}{}\doxysubsection{load\+All (string \mbox{[}, iterator\mbox{]} \mbox{[}, options \mbox{]})}\label{md_Backend_nodejs_node_modules_js_yaml_README_autotoc_md1648}
