You write code to describe your command line interface. Commander looks after parsing the arguments into options and command-\/arguments, displays usage errors for problems, and implements a help system.

\hypertarget{md_Backend_nodejs_node_modules_z_schema_node_modules_commander_Readme_autotoc_md3435}{}\doxysection{Options}\label{md_Backend_nodejs_node_modules_z_schema_node_modules_commander_Readme_autotoc_md3435}
Options are defined with the {\ttfamily .option()} method, also serving as documentation for the options. Each option can have a short flag (single character) and a long name, separated by a comma or space or vertical bar (\textquotesingle{}$\vert$\textquotesingle{}).

The parsed options can be accessed by calling {\ttfamily .opts()} on a {\ttfamily Command} object, and are passed to the action handler.

Multi-\/word options such as \char`\"{}-\/-\/template-\/engine\char`\"{} are camel-\/cased, becoming {\ttfamily program.\+opts().template\+Engine} etc.

An option and its option-\/argument can be separated by a space, or combined into the same argument. The option-\/argument can follow the short option directly or follow an {\ttfamily =} for a long option.


\begin{DoxyCode}{0}
\DoxyCodeLine{serve -\/p 80}
\DoxyCodeLine{serve -\/p80}
\DoxyCodeLine{serve -\/-\/port 80}
\DoxyCodeLine{serve -\/-\/port=80}

\end{DoxyCode}


You can use {\ttfamily -\/-\/} to indicate the end of the options, and any remaining arguments will be used without being interpreted.

By default options on the command line are not positional, and can be specified before or after other arguments.

There are additional related routines for when {\ttfamily .opts()} is not enough\+:


\begin{DoxyItemize}
\item {\ttfamily .opts\+With\+Globals()} returns merged local and global option values
\item {\ttfamily .get\+Option\+Value()} and {\ttfamily .set\+Option\+Value()} work with a single option value
\item {\ttfamily .get\+Option\+Value\+Source()} and {\ttfamily .set\+Option\+Value\+With\+Source()} include where the option value came from
